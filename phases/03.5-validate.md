# Phase 3.5: VALIDATE — Cross-LLM Technical Validation

> **Origin:** This phase was inspired by the Antigravity multi-agent video workflow
> (Prompt 3 pattern). Before dispatching to Build Agents, a dedicated Validation
> Agent verifies technical feasibility across all planned features.
>
> Catching an impossible architecture here costs 5 minutes.
> Catching it during Phase 6 costs days and requires a full rebuild.

## The Rule

No Build Agent executes until the Validation Agent signs off. Every planned feature — scroll engine, cursor system, page transitions, WebGL, audio TTS — gets verified against the actual browser environment, the chosen framework, and the performance budget.

---

## Validation Checklist — By Feature Area

### SCROLL ENGINE (Canvas + GSAP ScrollTrigger)

```
□ Canvas frame animation compatible with this framework (React/Next.js/Vanilla)?
  Note: React — use useRef for canvas, useEffect for GSAP. Next.js — ensure
  client-side only (dynamic import or 'use client')

□ Frame count × file size within 5MB total load budget?
  Calculation: [frame count] × [avg KB per frame] = [total KB]
  Budget: ≤ 5MB total for scroll sequence frames
  If over: Reduce count, lower resolution, or increase WebP compression

□ GSAP ScrollTrigger browser support verified?
  Support: Chrome 60+, Firefox 55+, Safari 12.1+, Edge 79+
  Mobile: iOS Safari 12.2+, Chrome Android 60+

□ Mobile: 30fps minimum achievable with frame count and resolution?
  Test: [frame_count] × [resolution] on low-end device estimate
  Fallback: Static hero if preload fails (required)

□ Preload progress indicator planned?
  Required: User must see loading progress before scroll-driven section activates
```

### WEBGL / THREE.JS

```
□ Project type eligible (Avant-garde score ≥ 8 AND Bold score ≥ 8)?
  If scores below threshold: Remove WebGL, use CSS/GSAP only

□ Mobile fallback designed?
  Required: Static image or simplified CSS fallback on touch devices
  Check: if ('ontouchstart' in window) skip Three.js init

□ renderer.dispose() cleanup planned on component unmount?
  Required for React/Next.js SPAs to prevent memory leaks

□ devicePixelRatio capped at 2?
  Required: Math.min(window.devicePixelRatio, 2) — always

□ Custom GLSL shaders — complexity appropriate for target performance?
  High risk: Fragment shaders with multiple uniforms on mobile GPU
```

### PAGE TRANSITIONS (Barba.js)

```
□ Site is multi-page (not SPA)?
  Barba.js: multi-page HTML only
  React/Next.js: Use Framer Motion AnimatePresence instead
  Next.js App Router: Custom transition with usePathname hook

□ Barba.js compatible with current routing system?
  Conflict risk: React Router, Next.js Router — verify compatibility
  Alternative: If SPA, use GSAP hooks on route change events
```

### CURSOR SYSTEM

```
□ Touch detection handled before cursor initialization?
  Required: if ('ontouchstart' in window) return;
  Also check: window.matchMedia('(pointer: coarse)')

□ Pointer Events API supported in target browsers?
  Support: All modern browsers ✓

□ Performance impact of continuous requestAnimationFrame loop assessed?
  Mitigation: Lerp-based cursor runs at screen refresh rate — acceptable
  Risk: Stacking multiple rAF loops — audit total rAF count
```

### AUDIO / TTS INTEGRATION (if planned)

```
□ WebSocket streaming TTS feasible in browser environment?
  Requirement: Secure context (HTTPS only)
  Browser restriction: WebSocket from browser requires HTTPS in production

□ AudioContext requires user gesture — handled?
  Required: AudioContext.resume() must be called inside a user event handler
  Pattern: Button click → audioContext.resume() → then speak()
  Autoplay restriction: Audio cannot play without prior user interaction

□ Scroll-sync narration latency budget achievable?
  Calculate: [TTS API average latency] + [audio buffer fill time] vs scroll trigger lead time
  If latency > 300ms: Add pre-trigger (fire TTS 300ms before scroll reaches trigger point)

□ Web Speech API as fallback?
  Available: Chrome, Edge, Safari (with vendor prefix)
  Not available: Firefox (limited support)
```

### TYPOGRAPHY / SPLITTEXT

```
□ GSAP SplitText requires GSAP Club/Business license?
  Free alternative: Split text manually or use custom split utility
  If budget restricted: Document the alternative implementation

□ Font loading strategy planned?
  Required: <link rel="preload"> for display font
  Required: font-display: swap on @font-face

□ Variable font — browser support verified for target audience?
  Support: Chrome 62+, Firefox 62+, Safari 11+
```
